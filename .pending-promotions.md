  Origin: 8 orphaned ndb-server children observed across multiple dashboard restarts in same session.

## 2026-05-28 Session — v2.3.0 ship: atoms IN nDB + N-ary contains + honest comparison

### Propose → ~/.claude/rules/programming.md
- Encoding width caps the capability pitch silently. If your design claims "supports N of X" (N-ary edges, N-dim vectors, N rows), verify the encoding holds a realistic N before shipping. u8 caps at 255, u16 at 65k. Pick width by domain upper bound + an order of magnitude.
  Origin: nDB HyperEdgeRecord arity encoded as u8 silently capped "N-dimensional" pitch at 255. KRAS protein_atoms hyperedge needed arity 1518 — blew up with ArityOverflow. Bumped to u32, format_version 1→2.
  Status: APPLIED-LOCAL 2026-05-28

- Indexed-extract from a blob → drop the blob. When you decompose a binary/text blob into queryable records, drop the blob. Two sources of truth always drift; you pay 2-3× storage for no read benefit. Reconstruct on demand if you need the original bytes.
  Origin: nDB stored both PROP_CIF_BYTES blob (180 KB) AND 1517 atom entities for KRAS. User caught the duplication. Path B: drop the blob, rebuild PDB in memory from atoms on subsequent loads.
  Status: APPLIED-LOCAL 2026-05-28

- KPI comparison hygiene — measure layer-by-layer, mark unmeasured. Three traps when comparing X vs Y: (a) mixed scopes (compare atomic-to-atomic, metadata-to-metadata), (b) per-item vs shared/amortized costs (schema dictionary is shared, not per-record), (c) speculative numbers — if you haven't benchmarked it, span both columns and say "not measured".
  Origin: User caught 3 misleading framings in a row during v2.3 explorer dev: "3 records vs 1847 rows", "180 KB blob vs 297 KB nDB", "16 MB schema dict per protein". Each fix improved trust.
  Status: APPLIED-LOCAL 2026-05-28

### Propose → ~/.claude/rules/frontend.md
- 3d-force-graph / d3-force never settle by default. cooldownTicks(N) + onEngineStop pinning fx/fy/fz + linkDirectionalParticles baseline = 0. Without all three, scene reads as constantly animated. Also static dev server must send Cache-Control: no-store or browser keeps stale HTML.
  Origin: User reported "3D dots jumping out" after multiple edits to particle config — browser was caching old HTML AND graph was never cooling. Three-line fix.
  Status: APPLIED-LOCAL 2026-05-28

- Test binaries that each spawn a server flake under whole-suite parallelism (`cargo test --workspace`, `pytest -n`): cross-binary port/resource contention, not a regression. Re-run the one test ISOLATED + in its own binary; green isolated + change doesn't touch that path ⇒ flaky-under-load. Bind ephemeral ports (`:0`).
  Origin: nDB — one ndb-server arrow test intermittently failed under workspace parallelism, passed isolated every time; /v1 + router changes never touched arrow.
  Status: APPLIED-LOCAL 2026-06-14

### Propose → ~/.claude/rules/release-publishing.md (NEW domain file, 6 rules)
- Cutting a versioned release (binaries matrix + container image + npm). Rules: (1) release cp binaries by `[[bin]]` name not crate name (a `foo-cli` crate builds `foo`); (2) monorepo — the tag is the SINGLE source of version truth, stamp it into every package + cross-package pin at publish (else a launcher@0.1.0 pins optionalDeps@0.1.0 while siblings are tag-stamped 2.4.0 → npx can't fetch the binary); (3) GHA secrets injected at JOB START — setting one mid-run only reaches later jobs; (4) npm `latest` = last-publish-wins not highest-semver → `npm dist-tag add pkg@X latest` after a publish race; (5) smoke-test the PUBLISHED package (`npm i`/`npx`/`docker run`), not the repo — release-consistency bugs are invisible to repo tests; (6) scoped npm publish needs the org to EXIST + account a member (token ≠ org); `npm whoami`+`npm org ls <scope>` diagnoses without CI.
  Origin: nDB v2.4.0 release — all six bit in one session (ndb-cli→ndb bin, 0.1.0/2.4.0 optionalDep drift, NPM_TOKEN set mid-run, latest race, smoke test caught it, @ndb org taken → n-dimension-database-ndb).
