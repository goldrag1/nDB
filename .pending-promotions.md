  Origin: 8 orphaned ndb-server children observed across multiple dashboard restarts in same session.
  Status: APPLIED-LOCAL 2026-05-27

## 2026-05-28 Session — v2.3.0 ship: atoms IN nDB + N-ary contains + honest comparison

### Propose → ~/.claude/rules/programming.md
- Encoding width caps the capability pitch silently. If your design claims "supports N of X" (N-ary edges, N-dim vectors, N rows), verify the encoding holds a realistic N before shipping. u8 caps at 255, u16 at 65k. Pick width by domain upper bound + an order of magnitude.
  Origin: nDB HyperEdgeRecord arity encoded as u8 silently capped "N-dimensional" pitch at 255. KRAS protein_atoms hyperedge needed arity 1518 — blew up with ArityOverflow. Bumped to u32, format_version 1→2.
  Status: APPLIED-LOCAL 2026-05-28

- Indexed-extract from a blob → drop the blob. When you decompose a binary/text blob into queryable records, drop the blob. Two sources of truth always drift; you pay 2-3× storage for no read benefit. Reconstruct on demand if you need the original bytes.
  Origin: nDB stored both PROP_CIF_BYTES blob (180 KB) AND 1517 atom entities for KRAS. User caught the duplication. Path B: drop the blob, rebuild PDB in memory from atoms on subsequent loads.
  Status: APPLIED-LOCAL 2026-05-28

- KPI comparison hygiene — measure layer-by-layer, mark unmeasured. Three traps when comparing X vs Y: (a) mixed scopes (compare atomic-to-atomic, metadata-to-metadata), (b) per-item vs shared/amortized costs (schema dictionary is shared, not per-record), (c) speculative numbers — if you haven't benchmarked it, span both columns and say "not measured".
  Origin: User caught 3 misleading framings in a row during v2.3 explorer dev: "3 records vs 1847 rows", "180 KB blob vs 297 KB nDB", "16 MB schema dict per protein". Each fix improved trust.
  Status: APPLIED-LOCAL 2026-05-28
