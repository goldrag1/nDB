chemistry_ndb

Chemical reactions as N-ary hyperedges — the example the whitepaper opens with, now actually built. ~43 curated reactions across organic synthesis, industrial inorganic chemistry, and two full metabolic pathways. And a demonstration of "living data": the middle pane splits into editor + live mirror, with propagation through nDB's /subscribe long-poll.

Launch live demo → PubChem (compound source)

What this is

  • ~43 reactions spanning Haber-Bosch, methane combustion, the Wacker process, Diels-Alder, Suzuki coupling, Wittig olefination, photosynthesis, cellular respiration, glycolysis (10-step), Krebs cycle (8-step), polyethylene + nylon-6,6 polymerisation, and the Belousov-Zhabotinsky oscillator.
  • 96 compound entities, 28 catalyst entities, 5 solvent entities. Compound SMILES + InChIKey + molecular formula are resolved from PubChem at seed-build time; results are cached.
  • One arity-N hyperedge per reaction, where N is the total of all reactant + product + catalyst + solvent role-fillers. Stoichiometry is encoded as repeated role-fillers — six CO₂ molecules in photosynthesis become six ROLE_REACTANT slots all pointing at the same CO₂ entity. No separate stoichiometry table, no junction rows.
  • Polyethylene polymerisation tops the arity at 1,003 (1,000 ethylene + 1 polymer + 1 catalyst + 1 solvent in one record).
  • 2 pathway entities with an ordered list of constituent reaction names, encoding the linear glycolysis chain and the cyclic Krebs loop.

The "living data" angle

You asked: does nDB support reactive data — edit a property, watch dependents update in real-time?

The honest answer: yes, via POST /subscribe long-polling + application-layer propagation. The engine itself doesn't have triggers or computed columns — that's by design (a database shouldn't conflate storage and compute). What nDB does provide:

PrimitiveHow it works
Append-only MVCCEvery write yields a new tx_id; prior versions stay in the log for audit + time-travel.
POST /subscribeLong-poll with since_tx_id; the server blocks on a condvar until any /commit lands a newer transaction. 30s default timeout, 60s max.
Manifest-based wakeCheap pre-check before blocking — if commits already landed past your cursor, return immediately.

The demo wires this end-to-end:

  1. The center pane splits 50/50: LEFT = editor (drag sliders, see local recompute), RIGHT = live mirror.
  2. Adjust the temperature slider on the LEFT. The SPA recomputes ΔG' = ΔG · (T₀/T) and K_eq' = exp(−ΔG'/RT) locally for instant visual feedback.
  3. Click Commit changes to nDB. The SPA POSTs a re-asserted REACTION hyperedge to /commit — same id, new property values. The engine assigns a new tx_id and notifies subscribers.
  4. The RIGHT pane has been hanging on a /subscribe call with since_tx_id = previous tx. The notification fires, the SPA re-fetches state, and every record in the RIGHT pane that shares any compound with the edited reaction flashes yellow.
  5. Watch the top-bar SUBSCRIBE pill — it's pulsing yellow and shows the current tx counter.

That's the same wire mechanism that would power multi-tab sync, multi-process pipelines, or downstream consumers in any reactive system. The application defines what "propagation" means; the engine delivers the change feed.

What this is not

  • Not chemically accurate for non-equilibrium conditions. The Van 't Hoff recompute uses ΔG' ≈ ΔG · (T₀/T) which assumes constant ΔH and zero ΔS — a deliberate simplification so the demo logic stays in ~5 LOC. Real chemists would want full thermodynamic tables and activity coefficients.
  • Not retrosynthesis. The pathway view is forward-only — glycolysis renders as 10 ordered steps, Krebs as 8-step cycle, but there's no "how do I make X" engine. (Considered for v2.)
  • Not a full reaction database. ~43 reactions are a curated story-telling sample; the Open Reaction Database has ~2M reactions and would be a target if we ever scale this up.
  • Not an engine-level "trigger" demo. nDB has no triggers — the recompute is JS. The demo proves the change-feed primitive that triggers would build on top of.

Schema

Entity types

  • T_COMPOUND = 1 — 96 compound entities with SMILES + formula + InChIKey
  • T_CATALYST = 2 — 28 catalysts (enzymes for biochem, transition metals for organic)
  • T_SOLVENT = 3 — 5 solvents (water, THF, DCM, hexane, diethyl ether)
  • T_PATHWAY = 4 — 2 pathway entities (Glycolysis, Krebs cycle) with an ordered list property of constituent reaction names

Hyperedge types

  • T_REACTION = 100 — arity 2–1003. Roles: ROLE_REACTANT (repeated per stoichiometric unit), ROLE_PRODUCT (same), ROLE_CATALYST, ROLE_SOLVENT.

Why pathways are entities not hyperedges

An obvious design would be T_PATHWAY as a hyperedge whose roles are the constituent reaction hyperedges in order. nDB doesn't support that today — hyperedge role-fillers must be entity ids (see JsonRole in crates/ndb-engine/src/wire.rs). So pathways are entities carrying a JSON-encoded ordered list of reaction names. Less "true N-ary" but pragmatic. Adding hyperedge-of-hyperedges to the engine is a future possibility — pathways would migrate naturally if it arrived.

cd nDB

python3 tools/chemistry/build_seed.py
