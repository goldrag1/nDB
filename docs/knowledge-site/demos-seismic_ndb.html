seismic_ndb

Earthquakes from the USGS Earthquake Catalog, with every aftershock sequence stored as one arity-(N+1) hyperedge. The 2010 Maule (Chile) sequence — mainshock plus 999 aftershocks — lives as a single record of arity 1000. Across the three shipped demos, that's the highest single-record arity in nDB.

Launch live demo → USGS Earthquake Catalog

⚠ The 4,774-event seed combines a 30-day live USGS snapshot with 13 curated historic mega-quake windows. The "live feed" button in the sidebar polls USGS's past-hour endpoint and commits any new events on the fly.

What this is

A single-page web app that lets you walk the world's seismicity the way nDB stores it:

  • 4,774 earthquake entities across two sources:
    • The 30-day USGS live feed (all_month.geojson) filtered to M ≥ 2.5 — the events of the current month.
    • 13 curated historic mega-quake aftershock windows (1989 Loma Prieta, 1995 Kobe, 2004 Sumatra-Andaman M9.1, 2008 Sichuan, 2010 Haiti, 2010 Maule, 2011 Tōhoku M9.1, 2015 Gorkha, 2017 Puebla, 2018 Anchorage, 2019 Ridgecrest, 2023 Türkiye-Syria, 2025 Myanmar Sagaing fault).
  • 17 AFTERSHOCK_SEQUENCE hyperedges: 4 detected algorithmically in the live feed (any M ≥ 6.0 event triggers grouping lower-magnitude events within 30 days + 200 km), 13 curated directly from the historic windows. Maule 2010's record has arity 1000; Tōhoku 2011's has arity 620; Sumatra 2004's has arity 453.
  • 14 major fault systems hand-curated with polyline traces: San Andreas, Hayward, Cascadia, Sumatran, Sunda, Japan Trench, Median Tectonic Line, North + East Anatolian, Dead Sea Transform, Alpine, Sagaing, Main Himalayan Thrust, East African Rift.
  • 108 ON_FAULT hyperedges linking events within ~30 km of a fault trace. Binary edges (the relation IS binary in this case — "this event happened on that fault") — proves nDB doesn't force every relation to be N-ary; arity follows the data.

The 30-second tour

  1. The map opens with all 4,774 events plotted on an equirectangular world projection. Color = depth (orange-red shallow → blue deep); radius = magnitude. The Pacific Ring of Fire appears immediately without any continents drawn — the dots are the structure.
  2. Drag the magnitude slider in the left sidebar. M ≥ 2.5 shows everything; M ≥ 6 reduces to ~30 globally-significant events of the month.
  3. Click any dot. The right pane shows the EARTHQUAKE entity, its AFTERSHOCK_SEQUENCE hyperedge (if it belongs to one), and the ON_FAULT hyperedge (if it's near a curated fault).
  4. Double-click a dot to jump the time window to its sequence's date range and select the mainshock. Useful for the historic mega-quakes — double-click any orange-outlined dot to focus that sequence.
  5. Pick a sequence from the top-bar dropdown. Sorted by arity descending — Maule 2010 (arity 1000) is at the top. Selecting it focuses the right pane on its mainshock and dims every other dot.
  6. Drag the time-window handles at the bottom to scrub through 1989–2026. The histogram shows event density; yellow bars are historic mega-quakes.
  7. Hit "Poll USGS" to fetch new events from the past hour. New events land as PLANET-style entities, the engine persists them, and the page refreshes.

Why "N-dimensional database" for this dataset

Earthquake sequences are a textbook N-ary fact. One mainshock spawns dozens to thousands of aftershocks, all sharing the same fault rupture, same time window, same epicentral region. A SQL representation would need:

TableRows
sequence(id, name, mainshock_event_id, window_days, radius_km, ...)1 per sequence
sequence_member(sequence_id, event_id, role)1000 for Maule alone
event(id, mag, depth, lat, lon, time, ...)1 per event

Querying "the Maule sequence" is a three-way JOIN; the relation between the mainshock and any individual aftershock is encoded as two rows in the junction table (one for mainshock, one for aftershock). In nDB, the sequence is the record. seq.roles is a flat list of 1,000 entries — 1 mainshock + 999 aftershocks — with role tags distinguishing the two. One read, no JOIN.

Schema

Entity types

  • T_EARTHQUAKE = 1 — 4,774 events with lat/lon/time/mag/depth/place + USGS catalogue id
  • T_FAULT = 2 — 14 curated faults with polyline trace stored as JSON
  • T_AGENCY = 3 — 14 reporting networks (us, ci, nc, ak, hv, …)

Hyperedge types

  • T_AFTERSHOCK_SEQUENCE = 100 — arity 2–1000: [mainshock, aftershock₁, …, aftershockₙ]
  • T_ON_FAULT = 101 — arity 2: [event, fault], with distance-to-fault as edge property
  • T_HISTORIC_EVENT = 102 — arity 1: [event]; a marker so the UI can filter "show only mega-quakes" without trawling every entity

Data provenance

Every event is identified by its USGS catalogue id (e.g. official20110311054624120_30 for Tōhoku) and traces back to the original USGS Comcat record. The 30-day live snapshot was pulled from earthquake.usgs.gov/earthquakes/feed/v1.0/summary/all_month.geojson; the historic windows from the FDSN event query API at earthquake.usgs.gov/fdsnws/event/1/query.

The aftershock-grouping heuristic for the live feed (M ≥ 6.0 → group lower-magnitude events within 30 days + 200 km) is approximate — published sequence boundaries from individual papers would be more precise, but the heuristic catches the dominant patterns and lets the demo extend to any new significant event automatically.

What this is not

  • Not a replacement for the USGS Catalog. Treat nDB as a working set; the catalog is the source of record.
  • Not full-catalogue. 4.7k events sampled to tell a story. The engine itself scales to millions; the SPA's SVG rendering would not — that's a visualisation limit, not an engine limit.
  • Not real-time streaming. "Live fetch" polls on demand, doesn't push.
  • Not seismically authoritative on aftershock grouping. The 30-day / 200-km heuristic mirrors what's published in declustered catalogues but ignores Omori decay subtleties.

Run it yourself

git clone https://github.com/goldrag1/nDB
cd nDB

# Regenerate the seed (re-fetches from USGS):
python3 tools/seismic/fetch_historic.py
curl -fsSL https://earthquake.usgs.gov/earthquakes/feed/v1.0/summary/all_month.geojson \
  -o .demo-data/source-data/seismic/all_month.geojson
python3 tools/seismic/build_seed.py

# Launch:
cargo run -p ndb-renderer --example seismic_explorer
# then open http://127.0.0.1:9878/

The engine state lives at <workspace>/.demo-data/seismic-ndb (project-local, gitignored). Events you add via the live-fetch button persist between runs.
