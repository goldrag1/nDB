langgraph_ndb

A GraphRAG-style knowledge graph, browsed in 3D. 2,500 machine-learning papers and 10,466 authors from OpenAlex (public-domain, CC0 metadata) ingested Rust-native straight into nDB — no Python scraper, no intermediate file. Where the other demos show how nDB stores awkward facts, this one shows how it answers: recursive citation traversal, semantic vector search, and time-travel, all from one engine, all wired to an interactive 3D graph.

The query-side demo

The structural-biology, exoplanet, seismic and ecology demos make one point: a single N-ary hyperedge models facts that relational tables have to shred. That's the storage story. This demo is the query story — the three capabilities a graph + vector + temporal workload needs at once, which normally means three systems:

You do this……which is this nDB feature…that a relational stack handles badly
Click a paper → trace its citation ancestry Recursive traversal (relation+, depth-bounded) over an in-RAM adjacency index WITH RECURSIVE re-joins the edge table at every hop — the workload nDB beats ~4× on the bench
Type a phrase → jump to the nearest papers Vector kNN (vector_search) over abstract embeddings, indexed natively Needs a separate vector DB; keeping it in sync with the graph is its own project
Drag the year slider → watch the field grow MVCC time-travel (as of a snapshot) — append-only history is the storage model, not a bolt-on Temporal tables / SCD-2 bookkeeping, or a full audit-log rebuild
See a paper's authors as one fact N-ary hyperedge — AUTHORED is one record: [paper, author₁ … authorₙ] A junction table; "the authorship" isn't a row you can point at

Every dimension you can move on screen is one of those — there's no separate analytics layer. The graph, the embeddings, the citation chains and the per-year history all live in the same nDB instance.

Where nDB does the work: the store and the engine

Two layers, one process — that's the whole point of this demo.

  • The database (how it's stored). One append-only log of entities and N-ary hyperedges. A paper's authorship is a single AUTHORED record holding the paper and all its authors — not a junction table — and each abstract embedding is a vector stored on the paper itself, not in a sidecar service. Because the store is MVCC append-only, every per-year ingest leaves the prior state intact: the history is the storage model, so there's nothing extra to keep in sync for time-travel. The graph, vector and B-tree indexes are all rebuilt from that one log.
  • The engine (how it's answered). Each interaction is one query against those indexes — no ETL between systems. Citation ancestry is a recursive relation+ walk over the in-memory adjacency index; the type-ahead's semantic ranking is a vector_search k-NN over the vector index; the year slider is a snapshot_read as-of a transaction. One engine serves graph traversal, vector retrieval and temporal queries together — the combination you'd otherwise assemble from a graph database, a vector database, and a temporal/audit layer, and keep in sync by hand.

The five dimensions

  • x / y / z — a 3D force layout over the CITES topology.
  • time — the "as of year" slider. Each paper was ingested in its publication-year transaction, so the slider is a real MVCC snapshot: a handful of papers in 2012, all 2,500 by 2026.
  • semantic — the Semantic layout toggle pins every node to a projection of its embedding, so position itself encodes meaning; clusters of related work pull together.
  • colour — research field. size — citation count.

Zoom: detail on demand

The view uses semantic zoom — what you see depends on how close you are, the way a map shows countries when you're far out and streets when you're close. Each node carries an importance score (papers by citation count, authors by how many papers they bridge), and the camera distance sets a visibility threshold:

  • Zoomed all the way out — only the ~19 field "galaxies" (cluster super-nodes) and the handful of all-time mega-hub papers. The whole field at a glance, a few dozen nodes.
  • Zooming in — the galaxies dissolve and papers fade in in importance order: the most-cited first, then the mid-tier, then the long tail. Citation arrows and titles switch on as the field around you thins out.
  • Closest — every paper near you plus its co-authors, with directional citation edges and labels.
  • Click a galaxy to dive straight into a field; zoom back out and it collapses to a single node again.

The point isn't cosmetic: because the threshold rises as you pull back, the number of things actually drawn stays roughly constant at every zoom level — a few hundred to a couple thousand — no matter how large the underlying graph is. That's what keeps the 3D view fluid while letting nDB hold far more than the screen could ever show: zoom is the query, and nDB answers with just the slice you need.

The 30-second tour

  1. Start typing in the search box. A type-ahead dropdown blends literal title/author matches (prefix-ranked) with semantic neighbours — your text is embedded with the same function nDB used to index every paper (byte-for-byte) and ranked by cosine. Arrow-key through them, Tab to accept the suggestion, or Enter to search your typed text; the graph lights up the nearest matches as you go.
  2. Click a paper. A details panel opens — field, year, citations, the author chips, the references it cites and the papers that cite it, all clickable to hop onward. The graph highlights its citation ancestry to depth 3; everything else fades to a dim twinkle.
  3. Drag the year slider back to 2015 and forward. Nodes and edges appear exactly as the field accreted — time-travel, not a filter on a static dump.
  4. Scroll to zoom out. The 2,500 papers collapse into ~19 field galaxies; scroll back in and they bloom into papers, then authors. The on-screen count stays bounded the whole way, so it never stutters.
  5. Flip the layout to Semantic. The citation tangle reorganises into meaning-space; papers that share vocabulary and field migrate together.

How it's built

A single Rust binary (tools/langgraph) fetches a connected, top-cited slice of the ML / deep-learning / transformer literature from OpenAlex over HTTP, keeps only the citations internal to the slice, and caches it as CC0 JSON committed to the repo. The default run ingests that cache directly through the embedded engine (begin_write / put_entity / put_hyperedge) — one transaction per publication year, so the time-travel timeline is honest — then exports the viz feed the 3D page reads. No Python, no seed.json round-trip: a demo of a Rust-native database, built in Rust. The explorer itself is one self-contained HTML file (3d-force-graph over WebGL), no build step.

Why 2,500 and not a million? The 3D view is the only limit, not the database. nDB ingests every paper and all 10,466 authors; the explorer renders the 2,500 papers plus the 1,490 co-authors that link two or more of them — the connective structure — and semantic zoom keeps only a bounded slice on screen at any moment. The engine itself runs comfortably at 49k+ entities and millions of ops/sec; the demo is sized for your GPU, not for nDB. Push the zoom-driven, bounded-view idea to its limit — fetch each slice live as you pan, like map tiles — and the same explorer rides on top of an arbitrarily large nDB.
