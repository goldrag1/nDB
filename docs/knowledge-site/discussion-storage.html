Does N-dimensional cost more storage than tables?

A common intuition: "if every entity stores its own property labels, they repeat across entities — that has to cost more than a table where the column name is stored once in the schema." The short answer is no, not in nDB — but only because nDB doesn't store label strings per entity. The trap is real; the design specifically dodges it.

The naive "schemaless tax"

The textbook critique of property-bag / document stores (BSON, JSONB, RDF-as-strings) is exactly this:

  • Table DB: schema stores column_name once. A row is just [v1, v2, v3] — position implies meaning. Per-row label cost = 0 bytes.
  • Naive property-bag: every entity stores {"name": "KRAS", "uniprot": "P01116", ...}. The strings "name", "uniprot" repeat in every entity. With N entities × K properties × ~10 bytes/label, you pay N × K × 10 bytes just for labels.

For 1M entities × 5 properties × 10-byte label = 50 MB of pure redundancy. That's the trap.

What nDB actually does

Look at the schema constants for the alphafold demo: PROP_NAME=30, PROP_UNIPROT=39, PROP_ATOM_X=63, etc. These aren't strings stored per entity — they're small integer codes (1–2 bytes) and the string ↔ code mapping lives in a schema dictionary that's amortized across the entire DB.

So the per-entity cost of a label in nDB is:

  • prop_id: 1–2 bytes (varint or u16) — the reference
  • Value bytes (same as the table would store anyway)

vs the table DB's 0 bytes per row for the label.

The overhead is a few bytes per property per entity, not "the full label string per entity." On a 1518-atom KRAS structure with ~10 properties per atom, that's ~15 KB of prop_id overhead — negligible against the coordinate floats.

Where nDB wins on storage

Tables pay a hidden tax that property-bags don't: NULLs in sparse columns. If Protein has 50 possible properties but a given protein only fills 8, a wide table stores 42 NULL slots (1 byte each + nullability bitmap). nDB stores nothing for absent properties.

For heterogeneous data (proteins with different annotations, atoms vs residues vs pathways all in one DB), the table approach either:

  • Wastes space with mostly-NULL wide tables, or
  • Fragments into N tables + JOIN cost

nDB's variable per-entity property set is genuinely cheaper for sparse / heterogeneous data.

Where tables still win

  • Dense, uniform schema (every row has every column populated): the per-property prop_id overhead is pure loss vs positional layout.
  • Columnar formats (Parquet, ClickHouse): column header stored once, values run-length / dictionary encoded per column. Strongest compression story for analytics workloads — beats both row-store tables and property-bags.
  • Fixed-shape numeric data (coordinates, time series): a float[N][3] array beats N entities × 3 properties × (prop_id + float) every time. Exactly why mmCIF's atom_site loop is more compact than nDB's atom-entity representation on disk — and why the explorer's KPI table honestly splits "atomic data" from "metadata + relations" rather than claiming nDB wins everywhere.

The interesting question hiding underneath

The cost comparison isn't really "labels repeated vs not." It's what's the unit of schema sharing, and what's the access pattern.

ApproachSchema unitBest for
Wide tablePer-table, positionalDense uniform rows, OLTP
Narrow table (EAV)Per-row entity_id+attr_id+valueSparse — but kills query perf
Columnar (Parquet)Per-column, with dict + RLEAnalytics on dense numeric
nDB (dict-coded props)Per-DB schema dictionaryHeterogeneous entities + relations
Naive JSON / BSONNone (labels per doc)Worst of all worlds for storage

nDB sits closer to the columnar / dictionary-coded end than the JSON end — the prop_id is the same trick a columnar engine uses for column references. The "labels repeat" critique applies to a strawman (string labels per entity), not to nDB's actual encoding.

The honest summary

nDB pays a small per-property reference tax (1–2 bytes) in exchange for variable schema and unified storage of entities + N-ary relations. For protein / atom / pathway data with 10+ entity types and N-ary containment, that tax is much cheaper than the JOIN / NULL / multi-table cost of forcing it into a relational shape. For a 1B-row uniform time series, a table or Parquet column wins.
