nDB — a database for N-dimensional reality

Most real-world events involve many participants at once. nDB stores them as one record each, instead of fragmenting them across join tables or multiple binary edges. This is the public introduction; byte-level layouts and MVCC semantics live in the design spec.

The problem: reality is multi-dimensional, databases are 2D

Relational databases were designed in the 1970s for tabular data — payroll rows, ledger entries, inventory lists. The relational model normalized data into rows and columns, and SQL gave the world a powerful query language that has lasted half a century. But the events modern applications care about are rarely 2D.

Three quick examples from different domains:

  • Biology. A gene-expression event involves the gene, its transcription factors, co-factors, the chromatin state of its locus, the cell type, the time point, and the resulting protein. Seven role-players in one atomic fact.
  • Construction engineering. A structural load test involves the component, its material spec, the load applied, the measured deflection, ambient temperature, the engineer performing the test, and the timestamp. Seven dimensions all required to make the result regulatory-quality.
  • Business operations. An approval event involves the document, the approver, the timestamp, the workflow used, and the outcome. Five participants in one atomic event.

All three are inherently N-ary. Each loses meaning when fragmented into a 2D table. To store the business approval in SQL you need at minimum: an approvals table with a synthetic primary key, foreign keys to users, documents, workflows, a separate audit_log table, and an ORM layer that reconstructs the event from these fragments at query time.

Property graphs (Neo4j, TigerGraph) help — they put relationships first — but their edges are still binary. A 5-arity fact requires reification: a stub node connected to 5 binary edges. The fact's atomicity is lost. Triple stores (RDF) handle N-ary via reification too. Hypergraph databases (TypeDB, HyperGraphDB) get N-ary right but require upfront schema declaration, which excludes the AI-extraction use case.

The category gap nDB targets: a hyperedge-native database that is also schemaless-friendly, optimized for AI / ERP / scientific workloads, and architecturally minimal so the ecosystem can extend it.

What nDB is

A production-grade hypergraph database engine in Rust. Five foundational commitments:

1. Hyperedge-native data model

Every fact in nDB is a hyperedge — a connection between any number of named role-players, with optional properties. No reification, no synthetic intermediate node, no foreign-key choreography.

The same primitive expresses facts from very different domains:

# Chemistry — an N-ary reaction
HyperEdge {
    type: "chemical_reaction"
    roles: {
        reactant_1:   Sodium,
        reactant_2:   Chlorine,
        product:      SodiumChloride,
        catalyst:     Water,
        temperature:  T_25C,
        environment:  Exothermic
    }
    properties: { yield_pct: 98.5, study_ref: "PubChem-117" }
}

# Construction engineering — a structural load test
HyperEdge {
    type: "load_test"
    roles: {
        component:        BEAM-A12,
        material:         ReinforcedConcrete_C30,
        load_applied_kN:  250,
        deflection_mm:    1.2,
        ambient_temp_C:   22,
        tested_by:        engineer_pham,
        date:             2026-05-26
    }
    properties: { pass: true, certified_per: "TCVN-5574-2018" }
}

# Business — an approval event
HyperEdge {
    type: "approval"
    roles: {
        document:   SO-001,
        approver:   Alice,
        timestamp:  2026-05-26T15:00,
        workflow:   fast-track,
        outcome:    approved
    }
    properties: { comment: "all conditions met" }
}

One atomic fact per event. One write. One traversal. No reification. Nested entity hierarchies (body → cells → proteins → amino acids; building → floor → room → component) are expressed as standard hyperedges with a contains relation type.

2. Schema as metadata, not a separate primitive

"Schema" in nDB is not a separate engine concept. It is the collection of metadata hyperedges that describe other hyperedges. Type assertions, constraints, index declarations, inference rules — all written with the same query DSL as any data. 

write type_def(name: "Customer",
               required_properties: ["customer_code", "name"])

write constraint(target_type: "Customer",
                 rule_expr: "matches(tax_id, '^[0-9]{10,13}$')")

Consequences: no CREATE TABLE, no ALTER TABLE, no migration ceremony. Schema evolves at the speed of data. AI agents extracting facts can write type_def hyperedges as they discover concepts. Schema queries use the same DSL as data queries — no separate INFORMATION_SCHEMA. Schema is MVCC-versioned, so "what did our schema look like last year?" is a normal time-travel query.

Apps choose enforcement strictness per type: schemaless, soft-validated, strictly-validated, or fully reasoned over. The same engine serves AI extraction (loose) and ERP audit (strict).

3. Append-only storage with MVCC

The storage core is append-only. Updates are new assertions superseding old ones. Deletions are tombstone assertions. Compaction removes old versions per per-type retention policies.

  • Provenance is free. The history is the storage.
  • MVCC is natural. Every write creates a new version; no separate version-tracking machinery.
  • Time-travel is a feature. as of 2025-12-31 match customer(id: ?c, balance: ?b) works natively.
  • Audit trail is free. Every assertion records who, when, and from what source.
  • LSM-friendly. Append-only maps cleanly to log-structured merge trees.
  • GDPR-compatible. Selective deletion via compaction supports right-to-be-forgotten.

Realistic ERP workload (100M facts/year, 5-year retention) lands at ~100 GB hot + ~$2/month cold archival. Roughly 3× the storage of an update-in-place database, in exchange for free audit, MVCC, and provenance.

4. Engine retrieves; slicers compute

The engine is small. It does: primary storage (append-only LSM, custom binary format), pattern matching / filtering / projection / time-travel, MVCC transactions (snapshot isolation by default, serializable per transaction), the index framework, and compaction.

The engine does not do aggregation, sorting, grouping, or computation. Those live in slicer crates — composable companion crates that build on the engine's retrieval primitives. A slicer is also a declarative projection from the N-dimensional hyperedge graph onto a K-dimensional visual space (grammar of graphics, extended from tabular to hypergraph).

5. Plugin framework everywhere

Indexes, slicers, renderers, vector similarity, full-text search, geographic indexes — all plugins. The engine ships the framework; the ecosystem ships the features. GPU acceleration arrives in v2 as opt-in plugin crates (vector indexes, columnar aggregation, GPU slicers) without engine changes.

Three worlds where this matters

AI agent reasoning and LLM knowledge graphs

LLMs naturally produce N-ary structured facts when given the freedom to. "Alice gave Bob a book on Tuesday in the library" is one mental fact for an LLM — and it's one hyperedge in nDB. Current GraphRAG pipelines work hard to bend LLM output into reified property-graph triples or constrained SQL schemas. nDB matches what LLMs produce natively. First- class MCP integration ships with v1: nDB-mcp-server exposes the engine as a Model Context Protocol server — Claude, GPT, Llama, any MCP-aware agent can drive nDB without custom integration glue.

Multi-party operational systems

BIM data is naturally hypergraph-shaped — components with many attributes and many relationships. 3-way invoice match (PO + GRN + Invoice) is one hyperedge, not three tables + a match table. International trade events are 5-party native. Per-tenant SaaS: each tenant defines its own Customer type without database sprawl. Regulatory time-travel: Vietnamese VAS transition from TT200/2014 to TT99/2025 with auditors in 2030 reconstructing 2024 state without backups.

Scientific, biomedical, provenance/lineage

Most experimental events involve many parameters. W3C PROV-O lineage facts are 6+-arity: (output, derived_from, inputs…, by_transformation, at_time, with_parameters, by_researcher). nDB stores these natively without reification. EUDR compliance: trace crop batches from harvest geolocation through processing to retail — one hyperedge per stage, atomic and multi-party.

Performance characteristics

Strong points:

  • Point lookups: comparable to RocksDB / LevelDB — sub-millisecond on hot data
  • Pattern traversal: comparable to property graphs for binary patterns; faster for N-ary because there are no reification hops
  • Time-travel queries: same cost as current-state queries (MVCC + snapshot reads)
  • Append throughput: single-writer + batching, RocksDB territory (10K+ writes/sec per writer)
  • Read concurrency: lock-free MVCC; many readers don't contend
  • Streaming subscriptions: append-only log makes change-feed trivial

Honest trade-offs:

  • Ad-hoc aggregation without prepared indexes: slower than DuckDB / ClickHouse / BigQuery. Mitigated by columnar index plugin (v2).
  • Multi-writer throughput in v1: single-writer + batching. Multi-writer revisited in v3+ alongside distribution.
  • Distributed transactions: v3+. v1 is single-node.

What nDB is not

  • Not a relational replacement for OLTP banking, payroll, and other workloads where SQL has 50 years of operational hardening. nDB is for the workloads SQL was not designed for.
  • Not a graph database in the property-graph sense (Neo4j, TigerGraph). Those are binary-edge engines. nDB is hypergraph-native.
  • Not a vector database by itself. Vector similarity is a v2 plugin index, alongside structured queries — not the engine's primary purpose.
  • Not a triple store. RDF triples are 3-arity; nDB hyperedges are N-arity. The engines target overlapping but distinct workloads.
  • Not distributed in v1. Single-node with strong durability and replication. Distribution arrives in v3.

Risks honestly

  • Adoption inertia. SQL is the world's default. Convincing a team to adopt a hypergraph database is a real cost. nDB is positioned for workloads where SQL is the bottleneck — not as a SQL replacement.
  • Operational maturity. v1 is foundational. Backup / restore / replication tooling lags Postgres by decades. Early adopters accept this.
  • Query language familiarity. nDB's DSL is new. Tooling, IDE support, ecosystem libraries — all need building. We bet on LLM-native query generation to compress this gap, but it's a bet.
  • The hyperedge vs binary debate isn't settled. Reasonable people disagree about whether N-ary primitives are worth the complexity vs reification on binary edges. nDB is taking the N-ary side; the evidence in its favor accumulates with every demo (see 5 examples).
