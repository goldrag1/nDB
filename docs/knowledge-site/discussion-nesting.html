How many ways are there to declare nested entities?

nDB has no literal "nested" record type. There is only Entity, HyperEdge, and Property. "Nesting" is always modeled. Three fundamental mechanisms combine into six distinct patterns you actually see in practice.

The three fundamental mechanisms

  1. HyperEdge — a record that names a set of entities together (arity 2 → ∞).
  2. Property on the child — child entity stores an attribute referencing the parent (by id, name, or position).
  3. Type + convention — implicit grouping (all entities of type T with prop X = Y belong together), no stored link.

Everything else is a combination.

The six patterns for "X contains Y"

1. N-ary contains hyperedge (v2.3 canonical)

One hyperedge of arity N+1: [container, member₁, …, memberₙ]. Roles distinguish the container slot from member slots.

  • Example: protein_atoms arity 1518 = 1 protein + 1517 atoms.
  • Best for: true 1-to-N containment where the set is the unit of meaning.
  • Cost: one record, one variable-length id list.

2. Binary "X_of" edges (v2.2 — now an anti-pattern)

N separate binary hyperedges, each [child, parent] with role tags.

  • Example: 1517 atom_of edges instead of one protein_atoms.
  • Pays for: per-edge id duplication of the parent, N records instead of 1.
  • The whole reason nDB exists is to replace this with #1. Still legal, just wasteful.

3. Property-as-foreign-key (lightweight link)

Child stores PROP_PARENT_ID = <entity_id> directly. No edge record.

  • Example: an atom could carry PROP_RESIDUE_ID pointing to a residue entity.
  • Best for: uniform 1-to-1 parent-child where you'd never query "list all children of P" without already touching the child.
  • Cost: zero edges, but no first-class "the set" record to attach properties to — no confidence score on the grouping, no role tags.

4. Property-as-logical-coordinate (convention link)

Child stores attributes that describe the parent without referencing its id. Reader resolves via query.

  • Example: an atom carries PROP_ATOM_RES_POS = 42 + PROP_ATOM_RES_NAME = "GLY" — no link to a residue entity, just enough info to reconstruct "which residue" by convention.
  • Best for: when the parent might not have its own entity (residues exist only logically in PDB-flat data).
  • Cost: no integrity guarantee; rename breaks the link.

5. Multi-level hyperedge composition (skip-level + per-level)

Two or more hyperedges at different granularities for the same hierarchy.

  • Example: protein_residues (arity 45) AND protein_atoms (arity 1518) on the same protein — skips the residue → atom level.
  • Or: protein_residues + per-residue residue_atoms edges — fully hierarchical.
  • Best for: when both "all atoms of protein P" and "all residues of protein P" are hot queries. Skipping a level avoids a JOIN-equivalent traversal.

6. Motif hyperedge (peer-group, parent implied)

A hyperedge over peer children where the parent isn't a slot — parent is inferred because all members share it.

  • Example: catalytic_triad arity 3 = 3 residues, all belonging to the same protein but the protein isn't in the edge.
  • alpha_helix arity 16 = 16 ordered residues.
  • Best for: functional groupings where the parent is recoverable from members and the grouping itself has meaning ("these 3 residues are a catalytic triad" is the fact).

How to choose

QuestionAnswer points to
Is the set a named thing with its own properties (confidence, role)?#1 N-ary contains
Is the child meaningful without the parent?#3 or #4 (property link)
Does the parent have its own entity record?yes → #1 / #3; no → #4
Are there multiple useful granularities of containment?#5 (multi-level)
Is the grouping a peer-relation, not container-relation?#6 (motif)
Are you writing N records for what is conceptually one fact?#2 — stop, refactor to #1

The underlying principle

The v2.2 → v2.3 rewrite (binary atom_of edges → one protein_atoms hyperedge) is the canonical lesson:

"Nested" should be one record per containment fact, not N records reifying the same fact N times.

The arity-N hyperedge is what makes nDB structurally different from a graph DB that only has binary edges — that's the whole pitch.

Patterns #2 and #4 still exist as escape hatches, but if you reach for them, you're either modeling something genuinely binary (#2: a single foreign-key relation between exactly two entities) or accepting that the parent doesn't deserve its own entity (#4).
