Benchmark report — nDB vs SQLite vs MariaDB

A head-to-head on one identical dataset and one host. The point isn't "nDB is fastest at everything" — it isn't. It's to show where an N-ary-native, embedded engine wins (storage, indexed lookups, concurrency, deep traversal under load) and where a mature relational planner still wins (single-thread joins) — measured, not asserted.

Methodology. Dataset: 50,000 entities (1,000 regions + 49,000 customers) and 50,000 N-ary facts (45,000 sales + 5,000 containment edges) — the same logical rows loaded into all three engines. nDB 2.4.0 (embedded, Rust) and SQLite 3.46 (embedded, via rusqlite — no GIL) are measured in-process by their Rust bench servers; each datapoint is the p50/p99 over 500 in-process iterations. MariaDB 10.11 is a client–server engine, measured end-to-end through a reused client connection over a local unix socket (1,500–4,000 iterations) — so its numbers include the client/protocol/socket cost an application actually pays. Embedded-vs-networked is a real axis, reported honestly, not hidden. Same Linux host; no figures are fabricated, and losses are shown.

4 · N-pattern join — cost vs join depth (k hops)

The same containment graph, walked to depth k — a k-way relationship. SQL expresses it as a recursive CTE (each level re-joins the edge table); nDB walks its adjacency index. Identical topology across engines; mean latency per query.

Depth knDBSQLiteMariaDB
2—¹79 µs150 µs
395 µs56 µs152 µs
4120 µs64 µs170 µs
597 µs66 µs209 µs
6107 µs77 µs268 µs

Lower = faster. The in-process engines (green/blue) stay flat as depth grows; only MariaDB (amber) climbs — each extra hop is another networked recursion level. nDB has no depth-2 point (engine recursion guard).

Honest reading. This dataset's containment graph is dense — the reachable set saturates within ~3 hops — so cost is bounded by node count, not depth. The two in-process engines stay flat and close; SQLite's recursive CTE is actually a touch faster than nDB's traversal here. The one clear depth effect is MariaDB climbing 150→268 µs as each extra level pays another networked round of the recursion. So join-depth isn't where nDB pulls ahead on this graph — its wins are storage, indexed lookups, and concurrency. ¹nDB's recursion guard rejected the depth-2 query (RecursionDepthExceeded) — a real engine limitation, reported not hidden. A sparse, high-fan-out graph (where each hop multiplies the intermediate result) is the test that would actually stress relational join-depth blow-up; that's a future dataset, not this one.

5 · Why the relational engines need more — the N-ary axis

An N-ary fact (a containment edge linking parent + child regions, a sale linking buyer + item) is one record in nDB. In SQL it's a junction-table row that repeats foreign keys, and answering "what's reachable in k hops" becomes a k-way join or a recursive CTE that re-scans the junction table at each level. That shows up twice in this report: in storage (1.8–2.6× more bytes for the normalised form) and in the traversal workloads, where nDB walks its adjacency index instead of re-joining a table. The deeper and more concurrent the access, the wider the gap.

6 · High-arity facts — storage vs arity (k roles per fact)

50,000 facts over 10,000 entities, each fact linking k entities by distinct roles. nDB stores each fact as one record (k refs). The standard relational model of a variable-arity relationship is an association table fact_role(fact_id, role, entity_id) — N×k rows. So every extra role adds 50,000 rows on the SQL side and a few bytes/record on nDB's.

7 · Time-travel — "as-of" reads over version history

10,000 entities each updated to 10 versions (100,000 row-versions retained), then read as of the first version. nDB does this as a native MVCC snapshot read; SQLite needs a hand-built SCD2 table (valid_from/valid_to); MariaDB uses system-versioned tables (FOR SYSTEM_TIME AS OF). SQL sides use integer keys (their favourable choice).

Honest reading. nDB's win here is latency + ergonomics: an as-of read is a first-class snapshot — ~9× faster than a SQLite SCD2 index lookup, ~150× faster than MariaDB's networked AS OF — and at 0.45 µs it's essentially as fast as reading the latest value (0.32 µs), with no schema design needed. Storage is not an nDB win: SQLite's integer-keyed SCD2 (2.7 MB) is more compact than nDB's UUID-based 6 MB. So: reach for nDB when you query history constantly and want it free + native; if you only keep history and rarely read it, a SCD2 table is smaller.

8 · Vector search — a capability, not a speed win

50,000 × 128-dim embeddings. nDB has native kNN (HNSW snapshot) co-located with the data, so you can do "similar AND property=X" in one engine. SQLite and MariaDB 10.11 have no vector type — the only option is to pull the data into a vector library or bolt on an extension.

Honest reading. The win is capability, not raw speed: nDB keeps vectors in the same store as everything else, so semantic search composes with property/graph filters and time-travel without a second system. But at this scale nDB's kNN (4.5 ms) is actually slower than an optimised BLAS brute-force (1.2 ms) — its vector path has clear optimisation headroom and isn't a drop-in for a dedicated ANN library yet. Reach for it to avoid a separate vector store and to filter by structure, not to win a raw-throughput race.

Takeaways — where nDB wins, and where it doesn't

A serious comparison shows both. nDB's real, measured wins:

  • Storage (scaling): most compact at every size — 1.8–2.6× vs SQLite/MariaDB, gap holds to 1M rows.
  • High-arity facts: storage grows ~1 MB/role vs 8–12 MB/role for SQL's association table — the gap widens with arity.
  • Indexed lookups: 7–55× faster (embedded + native indexes).
  • Concurrency: lock-free MVCC reads scale 4–21× under load; recursion flips to nDB.
  • Time-travel reads: as-of queries are native snapshots — 9–154× faster than SCD2 / system-versioning, and as fast as reading the present.
  • Vector search: native + co-located + filterable (SQLite/MariaDB have no vector type) — a capability win.

…and where it doesn't (measured, not hidden):

  • Single-thread joins / pattern scans: a mature relational optimiser (MariaDB/SQLite) wins.
  • Schema change: modern SQL does instant ADD COLUMN — nDB's edge is only no-lock + sparse storage.
  • Version storage: integer-keyed SQLite SCD2 (2.7 MB) is more compact than nDB's UUID-based 6 MB.
  • Raw vector throughput: nDB's kNN (4.5 ms) is slower than optimised BLAS brute-force (1.2 ms) at this scale — its value is co-location, not speed.
  • Dense-graph deep traversal: when the reachable set saturates, in-process SQLite matches or beats nDB.

Where nDB fits — by application & industry

Each fit below is justified by a measured result above, not a slogan. nDB shines when several of its wins compound — N-ary relationships + time-travel + co-located vectors + concurrent reads, all embedded in one binary.

ApplicationWhy nDB (evidence)Industries
Knowledge graphs & scientific data N-ary facts as one record + high-arity storage win + co-located vectors for similarity. (§1, §6, §8) Pharma / drug discovery, genomics, cheminformatics, scholarly/citation, materials, biodiversity
Audit, provenance & compliance Native time-travel: "what did this look like at time T", who-changed-what, append-only — 9–154× faster as-of reads, free. (§7) Finance/regtech, healthcare records, legal/contracts, supply-chain provenance, government
Fraud, risk & AML graphs An event links payer+payee+device+merchant+time as one N-ary fact; fast indexed lookups + concurrent analyst reads + entity-similarity vectors. (§2, §3, §6, §8) Banking, payments, insurance, marketplaces
AI-agent memory & RAG backends Heterogeneous facts + vectors + time-travel ("what did the agent know at step T") in one embedded store with a built-in MCP server — no separate vector DB + SQL + audit log. (§7, §8) AI products, agent platforms, internal copilots
360° entity / relationship views Sub-µs point + indexed lookups, high concurrency, N-ary relationships without junction-table sprawl. (§2, §3, §6) CRM, identity/IAM, customer-data platforms, social
Digital twins, asset & network graphs N-ary asset↔sensor↔event relationships + time-travel state history + concurrent reads, embeddable at the edge. (§3, §6, §7) Manufacturing/IIoT, telecom inventory, energy/utilities, logistics

Less ideal for (per the not-wins above): heavy analytical/OLAP with big hash-joins and aggregations (use a columnar warehouse); high-contention transactional OLTP with rich constraints (a mature RDBMS); plain 2-D tabular CRUD (SQLite/Postgres is simpler); and massive-scale ANN serving (a dedicated vector DB). nDB's sweet spot is relationship-rich, version-aware, mixed-modality data read concurrently from an embedded engine — not a drop-in for a data warehouse or a pure vector store.

Reproduce: the bench servers are cargo run --release --example bench_race -p ndb-engine (nDB) and cargo run --release -p bench-race-sqlite-rust (SQLite); the MariaDB harness loads the identical rows into InnoDB with matching indexes. Raw method + numbers are in the project's benchmark journal.
