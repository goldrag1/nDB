Live race — nDB vs Postgres

Pick a challenger: Postgres lives on a network socket. nDB lives in-process. Most of nDB's wins below are partly the cost PG pays to be a server.

What this bench actually measures. This is not "which engine is faster" — both engines work fine across all of these workloads, often within microseconds of each other. What the bench surfaces is where each engine's architectural bet pays off.

  • nDB bets on graph-shaped data + every index in RAM + MVCC for free time-travel reads. It wins on access patterns that match: keyed point lookups, typed property lookups, graph walks, count-by-type.
  • SQLite bets on tabular data + B-tree everywhere + 25 years of query-planner work. It wins on relational access patterns: table scans, joins, ad-hoc SQL filters.
  • Postgres makes the same bet as SQLite but runs as a separate process, so it pays a libpq + network tax on every query that nDB and SQLite skip by being embedded. That tax is real and shows up on micro-latency workloads.

The workloads below are grouped by access pattern, not by raw speed. If your real workload is dominated by one of these patterns, the bench tells you something. If it's a mix, the absolute numbers matter less than knowing where each engine lives.

Both sides loaded the same 100,000-record shape on startup: 50,000 entities (49k customers + 1k regions) + 50,000 hyperedges (45k sales junctions + 5k region-containment edges). The schemas are honest — the relational sides use the natural junction-table encoding with indexes on every join key and ANALYZE run before this page went live. Same Linux host, same kernel, no Docker, single-threaded.

Pick a workload, hit Race. Both backends run the workload N times in parallel and report wall-clock μs at the user-call boundary (p50, p99, throughput). The winner's card lights up. Numbers vary ~10-30% run-to-run — the gap is the signal, not the headline number. Rate-limited to one race per workload per 3 seconds.

Where nDB is NOT the right answer. If your domain is naturally tabular — accounting ledgers, inventory transactions, sales orders, master data tables that join by foreign key — SQLite or Postgres is the correct tool. Trying to model classic ERP entities as nDB hyperedges would be friction without benefit; the bench shows SQLite handily wins the join-shaped workloads even on equal language footing. nDB earns its keep on the graph-shaped subdomains embedded inside larger systems: bills of materials, approval chains, org hierarchies, lot/serial traceability, audit trails, dependency cascades. Use the right tool per access pattern.

Same 100k dataset. Now both backends are hammered by N concurrent client threads for a fixed duration; we report sustained throughput (RPS) and the latency distribution (p50 / p95 / p99 / p999). As of v3-final the in-process backend wraps the engine in a RwLock<Engine>: read workers take the read slot and genuinely parallelise on the host CPU; the single-writer model is preserved by the writer slot. Postgres uses a libpq connection pool with N concurrent connections + the planner can execute many reads in parallel.

Both architectures parallelise reads now. The race shows where each side's in-memory vs socket+planner cost dominates: nDB's BTreeMap memtable + index probes are so cheap that even on 64 clients the per-op latency stays in the microsecond floor; PG's per-query libpq + planner cost is the floor cost that amortises better under heavier per-op work like join-shaped queries.
