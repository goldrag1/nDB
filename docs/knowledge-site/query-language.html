Query language playground

Pattern matching over hyperedges, with property projection, multi-pattern joins via shared variables, recursive transitive closure, aggregations, and time travel. The editor below talks to the live biodiv_ndb engine — 146 species, 22 regions, 63 interactions, 6 food webs. Click an example to try it, edit freely, hit Run.

The demo engine is read-only via this page. The create / delete / set / merge clauses are valid query-language syntax — try pasting one and the server returns 403 with a clean error. Same engine binary supports writes for CLI users; only the public surface is locked down.

Cheat sheet

match type(role: term, ...) [as ?v]
Pattern atom. term is a variable ?v, a literal ("text", 42, true), or a UUID uuid:....
?v.property_name
Property projection — follows the bound UUID to the record, returns the property value.
match a(...) b(...) ...
Multiple patterns share variables; same name binds to the same value (join).
where ?x <= 6 and ?y = "Vietnam"
type+ · type* · type? · type{1,3}
Recursive transitive closure on a hyperedge type. Anchor must be a literal UUID or a variable bound by an earlier pattern.
as of tx_id 42
Snapshot the query at a past transaction.
return ?a, ?b.name limit N
create type(prop: val, role: ?v) as ?new
Declarative insert. New entity or hyperedge gets a fresh UUID, optionally bound to ?new for downstream return.
match … delete ?v
Tombstone every bound entity / hyperedge. The old record stays visible via as of.
match … set ?v.prop = term[, …]
Update via supersede — read the record, write a new assertion with the named property replaced.
merge type(prop: val, …) [as ?v]
Upsert. If a record with all the binding values exists, bind to it; else create + bind.
count() · sum(?v.prop) · avg / min / max
Aggregates. When any return item is an aggregate, every non-aggregate return becomes an implicit GROUP BY key.

How this works

Each Run click sends the query text verbatim as a POST /biodiv_ndb/api/query/text body. The server (one nDB instance per demo) lexes + parses the text, resolves names against its dictionary, plans the patterns into an operator tree, and executes against the engine. Results come back as {columns, rows, truncated} JSON.

Read-only public surface. The five demo engines are launched with Server::with_read_only(true) on their HTTP front. /commit, /flush, /compact, and any /query[/text] whose body contains a create, delete, set, or merge clause return HTTP 403 with {"error": "read_only", "detail": …}. The engine itself is unchanged — the same binary running locally without the flag executes those clauses normally.

The implementation lives in crates/ndb-query (lexer + parser + resolver) and crates/ndb-engine/src/query (planner + executor). The full grammar is in the working spec.

v1 limits worth knowing

  • Property access in where isn't supported yet — bind the property as a variable inside the pattern and compare in where: 
    match pollination(season_from: ?sf, season_to: ?st)
where ?sf <= 6 and ?st >= 6
return ?sf, ?st
  • No subqueries / CTEs — queries are flat. Use limit if a single pattern's cardinality is too high.
  • SET on roles is not supported — only on properties. Changing a hyperedge's role-filler requires delete + create.
