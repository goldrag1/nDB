Three questions that came up while explaining nDB to people who already know relational databases. Each is a conversation, not a definitive answer — pushback welcome.

These arguments, now measured

The storage question below isn't theoretical anymore — against SQLite & MariaDB on 50k N-ary facts:

1.8–2.6×more compact

Full benchmark report — and an honest list of where tables still win →

Does N-dimensional cost more storage than tables? →

The intuition: "if every entity stores its own property labels, labels repeat — that has to cost more than a table where the column name is stored once." Why this is true for naive property-bags, why nDB sidesteps it, and where tables still win.

How many ways are there to declare nested entities? →

nDB has no literal "nested" type. Containment is always modeled. Three fundamental mechanisms combine into six distinct patterns — when each is right, when one is an anti-pattern, and what the v2.2 → v2.3 rewrite of protein_atoms taught us.

Five things tables can't describe well →

Multi-subunit protein assemblies, chemical reactions with full context, frame-semantic events in language, multi-species coevolution, multi-party treaties. For each: the nDB shape, the SQL shape, and what tables actually lose.
