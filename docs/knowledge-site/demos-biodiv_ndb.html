biodiv_ndb

Ecology is inherently N-ary. A pollination event names a plant, a pollinator, a region, and a season; a food web names an entire ecosystem alongside 15+ species at once. Demo #5 stores all of it as hyperedges with temporal qualifiers — and serves a Wikipedia photo for every one of the 145 species so you can actually see what each role is.

What this is

A single-page web app that lets you walk the world's biotic interactions the way nDB stores them:

  • 145 species entities, every one with a Wikipedia-sourced photo. The taxonomy spans plants, animals, fungi, lichens, microbes, and algae — across all six kingdoms a biology textbook would care about.
  • 22 region entities — continents, biomes, oceans, ecosystems. Each holds a lat/lon centroid so the map view can pin it.
  • 57 binary/ternary interactions across four categories: 15 pollination, 15 non-pollination mutualism, 12 parasitism, 15 predation. Every record is a hyperedge of arity 3 — two species plus the region.
  • 6 food-web hyperedges, one per major ecosystem (Yellowstone, California kelp forest, Serengeti, Indo-Pacific coral reef, Arctic, Amazon rainforest). The Yellowstone food web tops arity at 18 (1 ecosystem + 17 member species) in a single nDB row.
  • Temporal qualifiers on the hyperedges — May–July for yucca/moth, year-round for tropical hummingbird/heliconia, August–October for bald eagle / sockeye salmon. The inspector renders these as a 12-month strip so you can see at a glance when each interaction is valid.

The angle this demo nails: temporal qualifiers

Each of the previous four demos showed a different facet of nDB's value — protein structure, N-ary discovery facts, time-clustered events, stoichiometry-as-repeated-roles. Biotic interactions add the next axis: a fact that's only true at certain times.

The yucca / yucca-moth obligate mutualism is famously time-restricted: only May–July, when yucca blooms. In a relational schema you'd model that as either a junction table with `start_month` and `end_month` columns (lossy: months 12→2 wraps badly), or by exploding the relation into per-month rows (huge cardinality). In nDB it's just two properties on the hyperedge: season_from = 5, season_to = 7. The query "what's pollinating in June?" filters edges where the month falls in [from, to].

Schema

Entity types

  • T_SPECIES = 1 — 145 species, properties: scientific_name · common_name · kingdom · family · life_form · photo_url · wiki_url.
  • T_REGION = 2 — 22 geographic/ecosystem regions with lat · lon · region_kind.

Hyperedge types

  • T_POLLINATION = 100 — arity 3. Roles (plant, pollinator, region). Properties: season_from · season_to · obligate? · note.
  • T_MUTUALISM = 101 — arity 3. Roles (species_a, species_b, region). Properties: subtype · obligate?.
  • T_PARASITISM = 102 — arity 3. Roles (host, parasite, region). Properties: transmission.
  • T_PREDATION = 103 — arity 3. Roles (predator, prey, region). Properties: season_from · season_to.
  • T_FOOD_WEB = 104 — arity 11–18. Roles (ecosystem, member_1, …, member_N). One record per ecosystem.

Hyperedges of mixed cardinality, one type

All four pairwise types share an arity of 3 (two participants + one region). Food webs alone vary — Arctic = 11 roles, Yellowstone = 18. nDB doesn't enforce a single arity per type; the type just labels the kind of relation. SQL would model these very differently (interactions as junction rows; food webs as either an `ecosystem_members` table with thousands of rows, or as a JSON blob on the ecosystem). nDB makes them the same kind of object.

The 30-second tour

  1. Species mode — 145 cards, each a photo + common name + binomial + family. Filter by life form on the left; type a search term to narrow.
  2. Click any species → auto-switch to Network mode. The main pane becomes an ego-graph of the selected species: the species at the centre, every interaction partner around it, edges colored by interaction kind, arrows for the directional relations (predation, parasitism, pollination).
  3. Click a neighbour in the network to re-focus on it — the graph re-lays out, the inspector swaps to the new species and its hyperedges.
  4. Map mode. 22 region pins sized by interaction count. The Indo-Pacific and Sub-Saharan Africa light up brightest — they pack the most coevolved relationships.
  5. Food-web mode. Pick an ecosystem from the chips — the SPA computes the trophic level of each member (producer → apex), lays photos onto rows by level, and runs a force simulation to space them out horizontally. Arrows are the predator → prey edges; a green edge marks the few mutualisms that overlap with members of the same web (clownfish ↔ anemone in the coral reef, cleaner wrasse ↔ grouper).

What this is not

  • Not GloBI scale. GloBI holds ~3 M verified interactions; we hold 63. We picked textbook examples a non-biologist would recognise (yucca + yucca moth, polar bear + ringed seal, sea otter + urchin keystone-predation) rather than crowdsourced volume.
  • No directional asymmetry on mutualism. Pollination and parasitism are directional (plant ≠ pollinator; host ≠ parasite). Mutualism stores both species but doesn't claim one is "primary" — both roles are symmetric.
  • Food webs don't store edge directionality. A real food web is a directed graph (who eats whom). For the demo we collapsed that into "membership in this ecosystem". The narrative note carries the trophic-cascade story; the data does not.
  • Wikipedia photos may not always be the canonical individual. Some pages show a juvenile, a different sex, or a captive specimen. The seed builder takes Wikipedia's pageimage verbatim and caches it; curating individual photos is a separate exercise.

Run it yourself

git clone https://github.com/goldrag1/nDB
cd nDB

# Regenerate the seed (Wikipedia photos are cached at tools/biodiv/.photo-cache.json):
python3 tools/biodiv/build_seed.py

# Launch:
cargo run -p ndb-renderer --example biodiv_explorer
# then open http://127.0.0.1:9881/
