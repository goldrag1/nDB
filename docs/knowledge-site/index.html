One record per fact, no matter how many things it involves.

nDB is a hyperedge-native database for relations that aren't binary. Where a table needs a join table and a graph DB needs N edges, nDB stores one arity-N record — keeping the set, its roles, and its properties as a single first-class fact.

What it looks like, in four flavours

Why this exists

Most of reality is N-ary. A reaction has reactants, products, catalysts, conditions. A protein complex has subunits with roles and stoichiometry. A sentence has an agent, a recipient, a theme, a time, a location. Tables decompose these into junction tables — losing the fact that they're one thing. Graph DBs decompose them into N binary edges — paying N× the storage to represent one fact.

nDB takes the N-ary record as the primitive. An Entity is a thing; a HyperEdge of arity N is one record naming N entities together with named roles; a Property is an attribute on either. That's the whole data model.

A generic core, applications on top

nDB is split in two, on purpose. The core is a reusable, domain-agnostic database — append-only LSM storage, MVCC time-travel, and the adjacency / type-cluster / property-B-tree / vector indexes, with a query executor over them. It knows nothing about proteins, earthquakes, or research papers.

Each application embeds that core and builds its own thin server on top — composing the core's primitives (ordered scans, bounded traversals, vector kNN, as of snapshots) into exactly the query shapes and caches its workload needs. The language-graph demo, for instance, runs a small app-layer server that turns "zoom level" into a bounded view query and keeps its own citation-sorted index — so the browser stays fluid over an arbitrarily large graph while the engine underneath stays general. That's the point of the split: one engine, many applications, each free to optimise its own access without forking the database. Generic capabilities live in the core; domain-specific composition lives in the app.
